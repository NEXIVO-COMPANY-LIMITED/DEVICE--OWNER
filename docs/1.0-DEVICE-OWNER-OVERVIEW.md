# 1.0 - Device Owner Overview

## Project Objective

The **objective** of this project is to deliver an **Android Device Owner application** that:

- Secures and controls **loan-backed or asset-backed devices** so the lender/operator can enforce policy, monitor status, and protect the asset until the loan is complete.
- Runs with **Device Owner** and **Device Admin** privileges so the app can apply system-level restrictions and survive user tampering within Android's security model.
- Integrates with a **backend** for registration, heartbeat, lock/unlock, tamper reporting, and deactivation so the fleet is manageable from one centralized location.

---

## What the Project Must Deliver

The project is required to deliver the following core features. Nothing in this list is optional for the base product:

| # | Deliverable | Description |
|---|-------------|-------------|
| 1 | **Device Registration** | Loan number entry, device data collection (IMEI, model, etc.), and backend registration returning a stable `device_id` for all future API calls. |
| 2 | **30-Second Heartbeat** | Sending device telemetry and status to the backend at a fixed 30-second interval; processing server responses (lock, deactivate, next_payment, tamper_indicators). |
| 3 | **Hard Lock** | Full kiosk-style lock (HardLockGenericActivity) on server command or tamper; persists across reboot. |
| 4 | **Soft Lock** | Dismissible overlay for payment reminder or SIM-change warning, without full kiosk lockdown. |
| 5 | **Tamper Detection** | Detecting developer options, USB debugging, bootloader unlock, root, SIM change, app removal, etc.; applying hard lock and reporting tamper events to the backend. |
| 6 | **Security Hardening** | Restricting factory reset, unknown sources, and other high-risk actions; optional STRICT mode (no ADB). |
| 7 | **Offline Resilience** | Queuing events (e.g. tamper, logs) when offline and syncing when the network returns. |
| 8 | **Deactivation** | Clean deactivation when the loan is complete (server or user trigger), returning the device to an unmanaged state. |
| 9 | **Auto-Update** | Optional self-update from a trusted source (GitHub Releases) using Device Owner privileges. |
| 10 | **Backend Integration** | APIs for registration, heartbeat, installation status, tamper, logs, and bugs as defined in [6.0-APIS.md](6.0-APIS.md). |

These deliverables define the **minimum** the project must achieve; additional features (e.g. FRP, Knox integration) extend this base.

## Purpose (How the App Works)

The app fulfills the objective by:

### Loan/Asset-Backed Device Control
- Devices are tied to a loan account
- App enforces security policies
- Reports device status to backend
- Supports remote lock/unlock
- Enables deactivation when loan is complete

### Security Hardening
- Restricts factory reset
- Blocks developer options
- Disables USB debugging
- Prevents unknown source installations
- Controls user account management

### Tamper Detection and Response
Detects and responds to:
- Bootloader unlock attempts
- Root detection
- Developer mode activation
- USB debugging enablement
- SIM card changes
- App removal attempts

Response actions:
- Apply hard lock immediately
- Report to backend
- Queue event if offline
- Log locally for audit

### Heartbeat and Sync
- Sends device data every 30 seconds
- Processes server instructions
- Handles lock/unlock commands
- Manages deactivation requests
- Syncs offline events when online

### Silent Operation
- Management messages suppressed
- Policy dialogs hidden
- No visible security prompts
- Seamless user experience
- Implemented via `CompleteSilentMode`

## Security Modes

### STANDARD Mode
**Purpose**: General loan devices with standard security

**Restrictions**:
- Factory reset: Blocked
- Safe boot: Blocked
- Developer options: Visible but limited
- USB debugging: Disabled
- User management: Restricted
- Unknown sources: Blocked

**Use Case**: Most loan-backed devices

### STRICT Mode
**Purpose**: High-value devices or sensitive deployments

**Restrictions**:
- All STANDARD restrictions, plus:
- Developer options: Hidden
- ADB: Completely disabled
- USB file transfer: Blocked
- Maximum lockdown

**Use Case**: Premium devices, high-risk deployments

### Configuration

Mode is stored in `device_owner_config.xml` and applied via:

```kotlin
val deviceOwnerManager = DeviceOwnerManager(context)
deviceOwnerManager.applyAllCriticalRestrictions()
```

## Key Constraints

### Minimum Requirements
- **Android Version**: API 21 (Android 5.0) minimum
- **Recommended**: API 26 (Android 8.0) or higher
- **Device Owner**: Must be set via provisioning

### Provisioning Methods
1. **QR Code**: During setup wizard (recommended)
2. **NFC**: Tap NFC tag during setup
3. **ADB**: For development/testing only

### Foreground Service
- Heartbeat runs inside `SecurityMonitorService`
- Maintains stable 30-second interval
- Works when app is in background or closed
- Uses `systemExempted` foreground service type

## Data Flow

### Registration Flow
```
User Opens App
    ↓
Enter Loan Number
    ↓
Collect Device Data
    ↓
POST /api/devices/mobile/register/
    ↓
Receive device_id
    ↓
Store Locally (Encrypted)
    ↓
Start Heartbeat
```

### Heartbeat Flow
```
Every 30 Seconds
    ↓
Collect Device Status
    ↓
POST /api/device/heartbeat
    ↓
Receive Server Response
    ↓
Process Commands (lock/unlock/deactivate)
    ↓
Update Local State
```

### Tamper Flow
```
Tamper Detected
    ↓
Apply Hard Lock
    ↓
Log Event Locally
    ↓
POST /api/device/tamper
    ↓
If Offline: Queue for Sync
```
