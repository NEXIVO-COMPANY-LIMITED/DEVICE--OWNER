## 19.0 DEVICE DATA ENCRYPTION AND DECRYPTION

This document explains, in simple English, how the app **encrypts** and **decrypts** sensitive data (device ID, registration data, local databases).

---

## 1. Components involved

- **`EncryptionManager`**: Creates encrypted `SharedPreferences` instances using Android Keystore.
- **`EncryptedPreferencesManager`** / **`SharedPreferencesManager`**: High-level helpers that read and write encrypted settings.
- **`DeviceIdProvider`**: Securely stores and retrieves the server `device_id` using encrypted preferences.
- **SQLCipher (`net.sqlcipher.database.SQLiteDatabase`)**: Encrypts local Room databases on disk.

All of these work together so that **at rest** (on storage), important data is encrypted, and is only decrypted **in memory** when the app reads it.

---

## 2. Encrypted SharedPreferences

### 2.1 How data is encrypted

- `EncryptionManager(context).getEncryptedSharedPreferences(name)` returns a `SharedPreferences` instance that:
  - Uses keys from the **Android Keystore**.
  - Stores values (for example `device_id`, loan number, lock state) in **encrypted form** on disk.
- When you call `.putString()` / `.getString()`:
  - The value is **encrypted before write**.
  - The value is **decrypted on read**, so your code only sees plain text.

You do **not** need to call any low-level encryption APIs in your code; this is handled by `EncryptionManager`.

### 2.2 Where device ID is stored

- `DeviceIdProvider.saveDeviceId(context, deviceId)`:
  - Stores `deviceId` into two encrypted preference files:
    - `device_data_secure` (primary)
    - `device_registration_secure` (backup)
  - Also mirrors the ID into **Device Protected Storage** on Android 7+ so that it can be read right after boot.
- `SharedPreferencesManager.saveDeviceId(deviceId)`:
  - Writes the same ID into its own encrypted preference file.
  - Calls `DeviceIdProvider.saveDeviceId()` so all locations stay in sync.

### 2.3 How decryption works

- `DeviceIdProvider.getDeviceId(context)`:
  - Reads from encrypted preferences using `EncryptionManager`.
  - If it finds an ID, it **decrypts** and returns it as a normal `String`.
- `SharedPreferencesManager.getDeviceId()`:
  - Tries protected prefs → normal prefs → `DeviceIdProvider.getDeviceId(context)`.
  - In all cases, the actual bytes on disk are encrypted; your code only handles decrypted values.

---

## 3. Encrypted local databases (SQLCipher)

### 3.1 How SQLCipher is loaded

- In `DeviceOwnerApplication.onCreate()`:
  - `SQLiteDatabase.loadLibs(this)` is called **synchronously**.
  - This loads the SQLCipher native libraries before any Activity touches the database.

### 3.2 How data is encrypted

- `DeviceOwnerDatabase` (Room) is configured to use SQLCipher.
- When Room opens the DB:
  - It provides a **secret (passphrase)** to SQLCipher.
  - SQLCipher uses that secret to **encrypt all database pages** on disk.
- As a result:
  - If you copy the `.db` file from the device, it looks like random bytes and cannot be read without the key.

### 3.3 How decryption works

- When the database is opened again:
  - SQLCipher is given the same secret.
  - SQLCipher **decrypts pages on the fly** as Room reads data.
- Your DAOs and entities (for example `CompleteDeviceRegistrationEntity`) always see **normal, decrypted values**:
  - Insert → SQLCipher encrypts and writes.
  - Query → SQLCipher decrypts and returns.

---

## 4. Device data flow (registration → storage → heartbeat)

1. **Registration response**  
   - Backend returns a **server `device_id`** in the registration API response.

2. **Secure storage**  
   - `DeviceIdProvider.saveDeviceId(context, deviceId)` saves the ID into:
     - Encrypted preferences (`device_data_secure`, `device_registration_secure`).
     - Protected storage mirror (Android 7+).
   - `DeviceRegistrationRepository.updateRegistrationSuccessByLoan(...)` stores the same ID into the **encrypted SQLCipher database** row for that loan.

3. **Reading for later use**  
   - Heartbeat:
     - `HeartbeatManager.validateAndGetDeviceId()` → `DeviceIdProvider.getDeviceId(context)` (decrypt) → sends to server.
   - Logs / bugs:
     - `ServerBugAndLogReporter` resolves current `device_id` using `DeviceIdProvider` / `SharedPreferencesManager`.
   - UI and other features:
     - Read current `device_id` via `SharedPreferencesManager` or `DeviceIdProvider`.

At every step, **only the in‑memory value is plain text**; data at rest remains encrypted.

---

## 5. Direct Boot and consistency repair

- On Android 7+:
  - `DeviceIdProvider.saveDeviceId()` also mirrors the ID into **Device Protected Storage**, so it is available during Direct Boot if the keystore allows it.
- `DeviceIdProvider.verifyAndRepairConsistency(context)`:
  - Reads the `device_id` from all known encrypted locations.
  - If it finds a value in one place but not another, it **repairs** the missing copy so that all stores are consistent.

This ensures that after reboots or partial data clears, the encrypted `device_id` remains usable and synchronized.

---

## 6. Key takeaways

- All sensitive values (device ID, registration status, lock state, etc.) are stored using **Encrypted SharedPreferences** and/or **SQLCipher**.
- **Encryption and decryption are automatic** for your Kotlin code:
  - You call normal getters/setters and DAOs.
  - Under the hood, data is encrypted before write and decrypted on read.
- The registration → storage → heartbeat path always uses the same **server `device_id`**, passed through these secure layers.
